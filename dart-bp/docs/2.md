# Registrations

## Key registration
This is the first step where all entities, investors, auditors, mediators, register their keys. This is not registration for an asset but a proof of knowledge of secret key for the public key. A simple implementation of this is a proof of knowledge of the discrete log in these 2 relations which can be done with a folklore sigma protocol (Schnorr):
$$
EK = ek.G_{Enc}, \quad AK = sk.G_{Aff}
$$

But since we allow users (think institutions) to onboard a large number of keys in a single txn, we use the batch Schnorr protocol from Fig. 2 of [this paper](https://iacr.org/archive/asiacrypt2004/33290273/33290273.pdf) (Fig. 2 has a typo in the last equation where $g^y$ should be $g^s$) and implement the key registration as:

1. Prover (user) wants to register the public keys $[(EK_1, AK_1), (EK_2, AK_2), ... (EK_n, AK_n)]$ and has the corresponding secret keys $[(ek_1, sk_1), (ek_2, sk_2), ..., (ek_n, sk_n)]$
2. Prover picks random:

   $$r_{enc} \stackrel{\$}{\leftarrow} \mathbb{Z}_p$$

   $$r_{aff} \stackrel{\$}{\leftarrow} \mathbb{Z}_p$$

3. Prover creates $T_{enc} = r_{enc}.G_{Enc}, \quad T_{aff} = r_{aff}.G_{Aff}$
4. Prover hashes $[(EK_1, AK_1), (EK_2, AK_2), ... (EK_n, AK_n)]$, $T_{enc}, T_{aff}$ to get a challenge $c \in \mathbb{Z}_p$
5. Prover creates responses as:

   $$s_{enc} = r_{enc} + \sum_{i=1}^n c^i.ek_i$$

   $$s_{aff} = r_{aff} + \sum_{i=1}^n c^i.sk_i$$

6. Verifier now checks:

   $$s_{enc}.G_{Enc} \stackrel{?}{=} T_{enc} + \sum_{i=1}^n c^i.EK_i$$

   $$s_{aff}.G_{Aff} \stackrel{?}{=} T_{aff} + \sum_{i=1}^n c^i.AK_i$$

Similarly, multiple encryption keys of auditors/mediators can be registered in one key registration proof by following the protocol above and ignoring $AK_i$ and corresponding relations

## Account registration
This process is used by an investor when it wants to start trading in an asset, i.e. be either a sender or receiver of that asset. It can only be done once by any public key for a particular asset and is done only for non-fee assets. The public key $PK_{Aff}$ must already be registered as per the above protocol
A successful completion of registration results in the following:

1. A state $State_0$ being inserted as a leaf in the accounts curve tree where
   $$
   State_0 = sk.G_{Aff} + asset\_id.G_3 + \rho.G_4 + \rho^2.G_5 + s.G_6 + s^2.G_7 + id.G_8 + ek.G_{Enc}
   $$
   $balance$ and $counter$ are 0 during registration
2. asset-id, public key $(asset\_id, AK(=sk.G_{Aff}), EK(=ek.G_{Enc}))$, are added to the Asset-account mapping
3. Nullifier $N=\rho.G_5$ is revealed by user and chain adds it to the nullifier set.
4. If asset has a trustee with public key $pk_T$, encryption of $s$ and $\rho$ for $pk_T$ is published on chain.

**Account creation**

1. User picks a nonce $ctr$, appends to the asset-id $at$ to get the combined value $at||ctr$. It then picks a random $rho\_randomness$ and passes it along with $at||ctr$ to Poseidon2 hash to get the nullifier secret key $\rho = Poseidon2(rho\_randomness, at||ctr)$. The nonce $ctr$ is used in case user wants to re-register (account recovery, etc), it can generate a new $\rho$
2. User picks a random:
   
   $$s \stackrel{\$}{\leftarrow} \mathbb{Z}_p$$
   
3. It now creates its initial account state $State_0 = sk.G_{Aff} + asset\_id.G_3 + \rho.G_4 + \rho^2.G_5 + s.G_6 + s^2.G_7 + id.G_8 + ek.G_{Enc}$.


### Protocol
Here the prover (investor) wants to prove following relations:

1. $State_0 = sk.G_{Aff} + at.G_3 + \rho.G_4 + \rho^2.G_5 + s.G_6 + s^2.G_7 + id.G_8 + ek.G_{Enc}$. This is equivalent to proving $State_0 = AK + EK + at.G_3 + id.G_8 + \rho.G_4 + \rho^2.G_5 + s.G_6 + s^2.G_7$ since public key $AK$, encryption key $EK$, asset-id $at$ and identity $id$ are revealed to the verifier. So only $\rho, \rho^2, s, s^2$ need to be proved.
2. $N = \rho.G_5$
3. $\rho = Poseidon2(rho\_randomness, at||ctr)$
4. $\rho^2 = \rho.\rho$
5. $s^2 = s.s$
6. $AK = sk.G_{Aff}$
7. $EK = ek.G_{Enc}$
8. If asset has associated $pk_T$, then ciphertexts $C_s$ and $C_\rho$ correctly encrypt $s$ and $\rho$ respectively. Since $s, \rho \in \mathbb{Z}_p$ (belong to scalar field of Pallas curve) and are large, 255-bit in our case, if they are just encrypted in exponent-Elgamal, the decryption will be impractical since the decryptor ($pk_T$) has to solve discrete log of a 255-bit value.
   Thus prover breaks each into small chunks and encrypts each chunk using exponent-Elgamal encryption. These chunks are small enough so that discrete log over them can be completed in reasonable time (few minutes). $CHUNK\_BITS$ denotes the bit size of each chunk and $NUM\_CHUNKS$ denotes the total number of chunks. In our implementation, $CHUNK\_BITS$ = 48 and $NUM\_CHUNKS$ = 6 since $\lceil {\frac{255}{48}} \rceil = 6$ and this means 6 chunks are sufficient to encode the 255-bit value
   Since Elgamal is homomorphic, these encryptions of chunks can be combined to get an encryption of the whole value which can then be used in a Sigma protocol to show its equality with the value used in account state $State_0$


Following describes the protocol assuming a $pk_T$ exists but its simple to omit corresponding parts of the proof if $pk_T$ does not exist.

**Instance**: $State_0, AK, EK, at, ctr, N, id, C_{s_i}, C_{\rho_i}, pk_T, G_{Aff}, G_{Enc}, G_i, H_i$

**Witness**: $sk, ek, \rho, rho\_randomness, s, s_{\mathrm{chunks}_i}, \rho_{\mathrm{chunks}_i}, r_{\mathrm{enc}_i}, r_{\rho\_\mathrm{enc}_i}$.

Above are the high level instance and witness values known before the protocol starts and don't include instance and witness created during protocol execution like the various blindings and $T$ values.

#### Prover
1. Prover first wants to prove knowledge of $\rho, \rho^2, s, s^2$ in the relation $$\rho.G_4 + \rho^2.G_5 + s.G_6 + s^2.G_7 = State_0 - AK - EK - at.G_3 - id.G_8$$. This is just relation 1 from above rearranged where values on RHS are public. Lets call these $D$ as $$D = AK + EK + at.G_3 + id.G_8$$
   So prover wants to prove knowledge of $\rho, \rho^2, s, s^2$ in $\rho.G_4 + \rho^2.G_5 + s.G_6 + s^2.G_7 = State_0 - D$ and will do with a Sigma protocol
2. Prover picks random $\rho_r, \rho^2_r, s_r, s^2_r \in \mathbb{Z}_p$ and creates
   $$T_{state} = \rho_r.G_4 + \rho^2_r.G_5 + s_r.G_6 + s^2_r.G_7$$
3. $N = \rho.G_5$ is the initial nullifier. Proves knowledge of $\rho$ in $N$, sharing the blinding $\rho_r$ with the above sigma protocol. Creates:
   $$T_{null} = \rho_r.G_5$$
4. And to prove knowledge of secret key in its affirmation key $AK$, it picks random $sk_r$ and creates $T_{pk} = sk_r.G_{Aff}$
5. Prover also wants to prove knowledge of encryption key $ek$ in $EK$ as 
   $$ek.G_{Enc} = EK$$
   It picks random $ek_r$ and creates 
   $$T_{EK} = ek_r.G_{Enc}$$
6. If $pk_T$ exists, break randomness $s$ in $NUM\_CHUNKS$ chunks with each chunk as $CHUNK\_BITS$ bit long and encrypt each using exponent Elgamal as:
   $$
   C_{s_i} = (C_{s_{i,0}}, C_{s_{i,1}}) = (r_{\mathrm{enc}_i}.G_{Enc}, r_{\mathrm{enc}_i}.pk_T + s_{\mathrm{chunks}_i}.G), \quad C_{s_i} \in \mathbb{G}^2_p
   $$
   where $C_{s_i}$ is ciphertext of $i$-th chunk of $s$, $s_{chunks}$ is the base-$2^{CHUNK\_BITS}$ representation of $s$ and it has $NUM\_CHUNKS$ digits and $r_{enc_i} \in \mathbb{Z}_p$ is the randomness for the encryption.
   Similarly, break $\rho$ into $NUM\_CHUNKS$ chunks and encrypt each:
   $$
   C_{\rho_i} = (C_{\rho_{i,0}}, C_{\rho_{i,1}}) = (r_{\rho\_\mathrm{enc}_i}.G_{Enc}, r_{\rho\_\mathrm{enc}_i}.pk_T + \rho_{\mathrm{chunks}_i}.G), \quad C_{\rho_i} \in \mathbb{G}^2_p
   $$
7. The prover also needs to prove that each chunk (of both $s$ and $\rho$) is a $CHUNK\_BITS$-bit value to avoid overflow in the base-$2^{\mathrm{CHUNK\_BITS}}$ representation digits (which would make discrete log impractical). For this the prover commits to all chunks of $s$ and $\rho$ in a single Bulletproofs vector commitment $C_{BP\_s\_\rho}$ and enforces range proof constraints for each chunk.
   $$
   C_{BP\_s\_\rho} = b'.H_0 + s_{chunks_0}.H_1 + ... + s_{chunks_{NUM\_CHUNKS-1}}.H_{NUM\_CHUNKS} + \rho_{chunks_0}.H_{NUM\_CHUNKS+1} + ... + \rho_{chunks_{NUM\_CHUNKS-1}}.H_{2 \cdot NUM\_CHUNKS}, \in \mathbb{G}_p
   $$
   Here $b'$ is blinding to the commitment added by Bulletproofs.
8. Taking a linear combination of the $C_{s_{i,1}}$ components of ciphertexts as below:
   $$Com_s = \sum_{i} 2^{i \cdot \mathrm{CHUNK\_BITS}} \cdot C_{s_{i,1}} = \sum_{i} 2^{i \cdot \mathrm{CHUNK\_BITS}} \cdot r_{\mathrm{enc}_i} \cdot pk_T + s \cdot G$$
   The prover uses a Sigma protocol to prove knowledge of the combined randomness $r_{s\_combined} \in \mathbb{Z}_p$ (where $r_{s\_combined} = \sum_{i} 2^{i \cdot \mathrm{CHUNK\_BITS}} \cdot r_{\mathrm{enc}_i}$) and $s$ in $Com_s$. Prover reuses the blinding $s_r$ for $s$ and creates:
   $$T_{s-combined} = (r_{s\_combined})_r.pk_T + s_r.G, \quad T_{s-combined} \in \mathbb{G}_p$$
   Similarly for $\rho$:
   $$Com_\rho = \sum_{i} 2^{i \cdot \mathrm{CHUNK\_BITS}} \cdot C_{\rho_{i,1}} = \sum_{i} 2^{i \cdot \mathrm{CHUNK\_BITS}} \cdot r_{\rho\_\mathrm{enc}_i} \cdot pk_T + \rho \cdot G$$
   $$T_{\rho-combined} = (r_{\rho\_combined})_r.pk_T + \rho_r.G, \quad T_{\rho-combined} \in \mathbb{G}_p$$
9. For proving knowledge of $r_{\mathrm{enc}_i}, s_{\mathrm{chunks}_i}$ the prover initializes Sigma protocols, 2 for each chunk. It first creates blindings and then commits to them as
   For $i$-th chunk of $s$, pick random $r_{(\mathrm{enc}_i)_r}, s_{(\mathrm{chunks}_i)_r} \in \mathbb{Z}_p$ and create $$T_{\mathrm{enc}_i} = r_{(\mathrm{enc}_i)_r}.G_{\mathrm{Enc}}, \quad T_{\mathrm{chunks}_i} = r_{(\mathrm{enc}_i)_r}.pk_T + s_{(\mathrm{chunks}_i)_r}.G$$, where $T_{\mathrm{enc}_i}, T_{\mathrm{chunks}_i} \in \mathbb{G}_p$.
   Similarly for each chunk of $\rho$, pick random $r_{(\rho\_\mathrm{enc}_i)_r}, \rho_{(\mathrm{chunks}_i)_r} \in \mathbb{Z}_p$ and create corresponding $T$ values.
10. For proving $\rho = Poseidon2(rho\_randomness, at||ctr)$, $\rho^2 = \rho*\rho$ and $s^2 = s*s$, prover uses Bulletproofs constraints for Poseidon2 and two multiplications. Let $C_{BP_\rho}$ be Bulletproof's commitments to these values:
   $$
   C_{BP_\rho} = b.H_0 + rho\_randomness.H_1 + \rho.H_2 + \rho^2.H_3 + s.H_4 + s^2.H_5, \in \mathbb{G}_p
   $$
   Here $b$ is blinding to the commitment added by Bulletproof.
11. Prover proves knowledge of above committed values and equality of these with other commitments as:
   Pick random $r_{BP_\rho} \in \mathbb{Z}_p$ and $r_{BP\_s\_\rho} \in \mathbb{Z}_p$ and create
   
   $$
   T_{BP_\rho} = r_{BP_\rho}.H_0 + rho\_randomness_r.H_1 + \rho_r.H_2 + \rho^2_r.H_3 + s_r.H_4 + s^2_r.H_5, \in \mathbb{G}_p
   $$
   
   $$T_{BP\_s\_\rho} = r_{BP\_s\_\rho}.H_0 + s_{(\mathrm{chunks}_0)_r}.H_1 + ... + s_{(\mathrm{chunks}_{NUM\_CHUNKS-1})_r}.H_{NUM\_CHUNKS} + $$
   $$\rho_{(\mathrm{chunks}_0)_r}.H_{NUM\_CHUNKS+1} + ... + \rho_{(\mathrm{chunks}_{NUM\_CHUNKS-1})_r}.H_{2 \cdot NUM\_CHUNKS}, \in \mathbb{G}_p$$

12. Prover hashes the following to create challenge $c$ as:
    $$c = Hash(State_0, AK, EK, at, ctr, N, id, C_{s_i}, C_{\rho_i}, pk_T, Com_s, Com_\rho, C_{BP_\rho}, C_{BP\_s\_\rho}, $$
    $$T_{state}, T_{null}, T_{s-combined}, T_{\rho-combined}, T_{enc_i}, T_{chunks_i}, T_{BP_\rho}, T_{BP\_s\_\rho}, T_{pk}, T_{EK})$$
13. Prover now creates responses for each sigma protocol as:
    $$resp_{state} = [\begin{matrix}
    \rho_r + \rho.c, & \rho^2_r + \rho^2.c, & s_r + s.c, & s^2_r + s^2.c\\
    \end{matrix}]$$
    $$resp_{pk} = sk_r + sk.c$$
    $$resp_{null} = \rho_r + \rho.c$$
    $$resp_{EK} = ek_r + ek.c$$
14. $$resp_{r\_s\_combined} = (r_{s\_combined})_r + r_{s\_combined} \cdot c$$
    $$resp_{r\_\rho\_combined} = (r_{\rho\_combined})_r + r_{\rho\_combined} \cdot c$$
    Note that since prover already created response for witness $s$ as $s_r + s.c$ in $resp_{state}$, it has both responses for $Com_s$. Similarly, the response for $\rho$ is shared with $Com_\rho$.
15. Creates responses for each chunk ciphertext of $s$:
    $resp_{s_{i,0}} = r_{(\mathrm{enc}_i)_r} + r_{\mathrm{enc}_i}.c$
    $resp_{s_{\mathrm{chunks}_i}} = s_{(\mathrm{chunks}_i)_r} + s_{\mathrm{chunks}_i}.c$
    Similarly creates responses for each chunk ciphertext of $\rho$:
    $resp_{\rho_{i,0}} = r_{(\rho\_\mathrm{enc}_i)_r} + r_{\rho\_\mathrm{enc}_i}.c$
    $resp_{\rho_{\mathrm{chunks}_i}} = \rho_{(\mathrm{chunks}_i)_r} + \rho_{\mathrm{chunks}_i}.c$
16. For relations, $C_{BP_\rho}, C_{BP\_s\_\rho}$, only the responses for witnesses $b, b'$ need to be created as the responses of other witnesses are already created.
    $$resp_b = r_{BP_\rho} + b.c$$
    $$resp_{b'} = r_{BP\_s\_\rho} + b'.c$$
17. The proof is
    $$(C_{BP_\rho}, C_{BP\_s\_\rho},$$
    $$T_{state}, T_{null}, T_{BP_\rho}, T_{BP\_s\_\rho}, T_{pk}, T_{EK},$$
    $$T_{s-combined}, T_{\rho-combined}, T_{enc_i}, T_{chunks_i},$$
    $$resp_{state}, resp_{pk}, resp_{null}, resp_{EK}, resp_{r\_s\_combined}, resp_{r\_\rho\_combined}, resp_{s_{i,0}}, resp_{s_{chunks_i}}, resp_{\rho_{i,0}}, resp_{\rho_{chunks_i}}, resp_b, resp_{b'})$$
    
#### Verifier

1. Verifier hashes the following to create challenge $c$ as:
   $$c = Hash(State_0, AK, EK, at, ctr, N, id, C_{s_i}, C_{\rho_i}, pk_T, Com_s, Com_\rho, C_{BP_\rho}, C_{BP\_s\_\rho}, $$
   $$T_{state}, T_{null}, T_{s-combined}, T_{\rho-combined}, T_{enc_i}, T_{chunks_i}, T_{BP_\rho}, T_{BP\_s\_\rho}, T_{pk}, T_{EK})$$
2. Enforces constraints in Bulletproof for Poseidon2 for $\rho$ generation, and multiplications $\rho^2 = \rho.\rho$ and $s^2 = s.s$
3. Enforces constraints in Bulletproof for $2 \cdot NUM\_CHUNKS$ range proofs, $NUM\_CHUNKS$ for each of $s$ and $\rho$ chunks.
4. Verifies correctness of $State_0$ by checking:
   $$resp_{state}[0].G_4 + resp_{state}[1].G_5 + resp_{state}[2].G_6 + resp_{state}[3].G_7 \stackrel{?}{=} T_{state} + (State_0 - D).c$$
   $$\implies (\rho_r + \rho.c).G_4 + (\rho^2_r + \rho^2.c).G_5 + (s_r + s.c).G_6 + (s^2_r + s^2.c).G_7 \stackrel{?}{=} T_{state} + (State_0 - D).c$$
5. Verifies correctness of $pk$ by checking:
   $$resp_{pk}.G_{Aff}\stackrel{?}{=} T_{pk} + AK.c$$
   $$\implies (sk_r + sk.c).G_{Aff} \stackrel{?}{=} sk_r.G_{Aff} + AK.c$$
6. Verifies correctness of $N$ by checking:
   $$resp_{null}.G_5 \stackrel{?}{=} T_{null} + N.c$$
   $$\implies (\rho_r + \rho.c).G_5 \stackrel{?}{=} \rho_r.G_5 + N.c$$
7. Verifies encryption key relation:
   $$resp_{EK}.G_{Enc} \stackrel{?}{=} T_{EK} + EK.c$$
   $$\implies (ek_r + ek.c).G_{Enc} \stackrel{?}{=} ek_r.G_{Enc} + EK.c$$
8. Verifier creates $Com_s$ and $Com_\rho$ as:
   $$Com_s = \sum_{i} 2^{i \cdot \mathrm{CHUNK\_BITS}} \cdot C_{s_{i,1}}$$
   $$Com_\rho = \sum_{i} 2^{i \cdot \mathrm{CHUNK\_BITS}} \cdot C_{\rho_{i,1}}$$
9. Verifies correctness of $Com_s$ by checking:
   $$resp_{r\_s\_combined}.pk_T + resp_{state}[2].G \stackrel{?}{=} T_{s-combined} + Com_s.c$$
   Similarly for $Com_\rho$:
   $$resp_{r\_\rho\_combined}.pk_T + resp_{state}[0].G \stackrel{?}{=} T_{\rho-combined} + Com_\rho.c$$
10. Verifier checks response for each $s$ chunk ciphertext $C_{s_{i,0}}$:
    $$resp_{s_{i,0}}.G_{Enc} \stackrel{?}{=} T_{enc_i} + C_{s_{i,0}}.c$$
11. Verifier checks response for each $s$ chunk ciphertext $C_{s_{i,1}}$:
    $$resp_{s_{i,0}}.pk_T + resp_{s_{\mathrm{chunks}_i}}.G \stackrel{?}{=} T_{chunks_i} + C_{s_{i,1}}.c$$
    Analogous checks are done for each $\rho$ chunk ciphertext.
12. Verifier checks response for $C_{BP_\rho}$ as:
    $$resp_b.H_0 + resp_{state}[0].H_2 + resp_{state}[1].H_3 + resp_{state}[2].H_4 + resp_{state}[3].H_5 \stackrel{?}{=} T_{BP_\rho} + C_{BP_\rho}.c$$
    Note: responses for $rho\_randomness$ are provided as part of $resp_b$ (the partial response), while the responses for $\rho, \rho^2, s, s^2$ are reused from $resp_{state}$.
13. Verifier checks response for $C_{BP\_s\_\rho}$ as:
    $$resp_{b'}.H_0 + resp_{s_{\mathrm{chunks}_0}}.H_1 + ... + resp_{s_{\mathrm{chunks}_{NUM\_CHUNKS-1}}}.H_{NUM\_CHUNKS} + resp_{\rho_{\mathrm{chunks}_0}}.H_{NUM\_CHUNKS+1} + ... + resp_{\rho_{\mathrm{chunks}_{NUM\_CHUNKS-1}}}.H_{2 \cdot NUM\_CHUNKS} \stackrel{?}{=} T_{BP\_s\_\rho} + C_{BP\_s\_\rho}.c$$
14. Verifier finally checks the Bulletproofs proof that verifies all constraints, i.e for Poseidon2, multiplicationsHUNKS} + resp_{\rho_{\mathrm{chunks}_0}}.H_{NUM\_CHUNKS+1} + ... + resp_{\rho_{\mathrm{chunks}_{NUM\_CHUNKS-1}}}.H_{2 \cdot NUM\_CHUNKS} \stackrel{?}{=} T_{BP\_s\_\rho} + C_{BP\_s\_\rho}.c$$
14. Verifier finally checks the Bulletproofs proof that verifies all constraints, i.e for Poseidon2, multiplications and range proofs


## Encrytion key distribution
When an asset has a shared encryption key, the asset issuer distributes the corresponding secret key $ek$ to the relevant auditors/mediators using Twisted Elgamal and a key-distribution proof which proves to the chain that issuer has indeed encrypted the secret key for the intended recipients and they can decrypt to get it. The issuer splits $ek$ into chunks $ek_i$ and for each chunk produces:
- A shared ciphertext: $C_i = G_{Enc}.r_i + H.ek_i$
- A per-recipient ciphertext: $E_{ji} = EK_j.r_i$ for each recipient $j$ with encryption public key $EK_j$

Each recipient $j$ decrypts its chunk as $ek_i = C_i - ek_j^{-1}.E_{ji}$ and recovers the discrete log of $C_i - E_{ji}.ek_j^{-1}$ over $H$. The proof proves all chunks are in range, all randomnesses are consistent (same $r_i$ used in shared and per-recipient ciphertexts), and the chunks reconstruct the full $ek$. Twisted Elgamal is used to do the range proof on each chunk only once and not per recipient.

### Protocol
Here the prover (issuer) wants to prove following relations:

1. $C_i = G_{Enc}.r_i + H.ek_i$ for each chunk $i$ (correct shared ciphertext)
2. $E_{ji} = EK_j.r_i$ for each chunk $i$ and recipient $j$ (correct per-recipient ciphertext, using same $r_i$)
3. $pk = G_{Enc}.ek$
4. $ek_i \in [0, 2^{CHUNK\_BITS})$ for each chunk $i$ (range correctness)
5. The chunks $ek_i$ reconstruct to $ek$ committed in $pk$: $\sum_i 2^{i \cdot CHUNK\_BITS}.ek_i = ek$

**Instance**: $pk~(= G_{Enc}.ek), EK_1, ..., EK_n, G_{Enc}, H$

**Witness**: $ek, ek_0, ..., ek_{NUM\_CHUNKS-1}, r_0, ..., r_{NUM\_CHUNKS-1}$

Above are the high level instance and witness values. The secret key satisfies $ek = \sum_{i} 2^{i \cdot CHUNK\_BITS}.ek_i$.

#### Prover
1. Split $ek$ into $NUM\_CHUNKS$ chunks of $CHUNK\_BITS$ bits each: $ek = \sum_{i} 2^{i \cdot CHUNK\_BITS}.ek_i$.

2. For each chunk $i$, pick random $r_i \in \mathbb{Z}_p$ and compute:
   - Shared ciphertext: $C_i = G_{Enc}.r_i + H.ek_i$
   - Per-recipient ciphertexts: $E_{ji} = EK_j.r_i$ for each recipient $j$

3. For each chunk $i$, to prove $C_i = G_{Enc}.r_i + H.ek_i$, pick random $r_{i,r}, ek_{i,r} \in \mathbb{Z}_p$ and create:
   $$T_i = G_{Enc}.r_{i,r} + H.ek_{i,r}$$
   Note: The response for $ek_i$ will be shared with the Bulletproof vector commitment Schnorr protocol (step 6).

4. For each chunk $i$ and recipient $j$, to prove $E_{ji} = EK_j.r_i$, reuse the same blinding $r_{i,r}$ from step 3 and create:
   $$T_{ji} = EK_j.r_{i,r}$$
   Sharing $r_{i,r}$ cryptographically binds the same $r_i$ in both the shared and per-recipient ciphertexts.

5. Compute the combined randomness $r_{combined} = \sum_{i} 2^{i \cdot CHUNK\_BITS}.r_i$ and the combined ciphertext:
   $$Com_{ek} = G_{Enc}.r_{combined} + H.ek = \sum_{i} 2^{i \cdot CHUNK\_BITS}.C_i$$
   To prove knowledge of $r_{combined}$ and $ek$ in $Com_{ek}$, pick random $r_{combined,r}, ek_r \in \mathbb{Z}_p$ and create:
   $$T_{combined} = G_{Enc}.r_{combined,r} + H.ek_r$$

6. To prove $pk = G_{Enc}.ek$, reuse blinding $ek_r$ from step 5 and create:
   $$T_{pk} = G_{Enc}.ek_r$$
   Note: $ek_r$ is reused for proving $pk$ since the witness is the same $ek$.

7. Prover also needs to prove that each chunk $ek_i$ is a $CHUNK\_BITS$-bit value to avoid overflow in the base-$2^{CHUNK\_BITS}$ representation. For this it commits to all chunks in a Bulletproofs vector commitment and enforces range proof constraints for each chunk $ek_i$:
   $$C_{BP} = b.H_0 + ek_0.H_1 + ek_1.H_2 + ... + ek_{NUM\_CHUNKS-1}.H_{NUM\_CHUNKS} \in \mathbb{G}$$
   Here $b$ is the blinding added by Bulletproofs.

8. To bind the BP commitment to the sigma protocols, pick random $b_r \in \mathbb{Z}_p$ and create the Schnorr commitment, reusing $ek_{i,r}$ from step 3 for each chunk $i$:
   $$T_{BP} = b_r.H_0 + ek_{0,r}.H_1 + ek_{1,r}.H_2 + ... + ek_{(NUM\_CHUNKS-1),r}.H_{NUM\_CHUNKS}$$
   Sharing $ek_{i,r}$ ties the $ek_i$ committed in $C_{BP}$ to the $ek_i$ used in the shared ciphertext $C_i$.

9. Prover hashes the following to create challenge $c$:
   $$c = Hash(\text{nonce},~pk,~EK_1, ..., EK_n,~C_0, ..., C_{NUM\_CHUNKS-1},~T_0, ..., T_{NUM\_CHUNKS-1},$$
   $$E_{10}, ..., E_{n,NUM\_CHUNKS-1},~T_{10}, ..., T_{n,NUM\_CHUNKS-1},$$
   $$Com_{ek},~T_{combined},~T_{pk},~C_{BP},~T_{BP})$$

10. Prover now creates responses:
    $$resp_{r_i} = r_{i,r} + r_i.c \quad \text{(for each chunk } i\text{)}$$
    Note that $resp_{r_i}$ is reused for proving each $E_{ji}$ (step 4) since the witness is the same $r_i$.

    Responses $resp_{ek_i}$ for each chunk come from the Bulletproof Schnorr protocol:
    $$resp_{BP} = [b_r + b.c,\ \ ek_{0,r} + ek_0.c,\ \ ek_{1,r} + ek_1.c,\ \ ...,\ \ ek_{(NUM\_CHUNKS-1),r} + ek_{NUM\_CHUNKS-1}.c]$$

    $$resp_{r_{combined}} = r_{combined,r} + r_{combined}.c$$
    $$resp_{ek} = ek_r + ek.c$$
    Note that $resp_{ek}$ is reused for proving $pk$ (step 6) since the witness is the same $ek$.

#### Verifier

1. Verifier hashes the following to create challenge $c$:
   $$c = Hash(\text{nonce},~pk,~EK_1, ..., EK_n,~C_0, ..., C_{NUM\_CHUNKS-1},~T_0, ..., T_{NUM\_CHUNKS-1},$$
   $$E_{10}, ..., E_{n,NUM\_CHUNKS-1},~T_{10}, ..., T_{n,NUM\_CHUNKS-1},$$
   $$Com_{ek},~T_{combined},~T_{pk},~C_{BP},~T_{BP})$$

2. Enforces range proof constraints in Bulletproofs for each chunk $ek_i$ to be in $[0, 2^{CHUNK\_BITS})$.

3. For each chunk $i$, verifies the shared ciphertext using $resp_{BP}[i+1]$ as $resp_{ek_i}$:
   $$resp_{r_i}.G_{Enc} + resp_{ek_i}.H \stackrel{?}{=} T_i + C_i.c$$
   $$\implies (r_{i,r} + r_i.c).G_{Enc} + (ek_{i,r} + ek_i.c).H \stackrel{?}{=} G_{Enc}.r_{i,r} + H.ek_{i,r} + C_i.c$$

4. For each chunk $i$ and recipient $j$, verifies the per-recipient ciphertext using $resp_{r_i}$ from step 3:
   $$resp_{r_i}.EK_j \stackrel{?}{=} T_{ji} + E_{ji}.c$$
   $$\implies (r_{i,r} + r_i.c).EK_j \stackrel{?}{=} EK_j.r_{i,r} + E_{ji}.c$$

5. Verifies the Bulletproof Schnorr commitment (binding $C_{BP}$ to the sigma protocols):
   $$resp_{BP}[0].H_0 + resp_{ek_0}.H_1 + resp_{ek_1}.H_2 + ... + resp_{ek_{NUM\_CHUNKS-1}}.H_{NUM\_CHUNKS} \stackrel{?}{=} T_{BP} + C_{BP}.c$$
   $$\implies (b_r + b.c).H_0 + (ek_{0,r} + ek_0.c).H_1 + ... \stackrel{?}{=} b_r.H_0 + ek_{0,r}.H_1 + ... + C_{BP}.c$$

6. Computes the combined ciphertext:
   $$Com_{ek} = \sum_{i} 2^{i \cdot CHUNK\_BITS}.C_i$$

7. Verifies the combined ciphertext:
   $$resp_{r_{combined}}.G_{Enc} + resp_{ek}.H \stackrel{?}{=} T_{combined} + Com_{ek}.c$$
   $$\implies (r_{combined,r} + r_{combined}.c).G_{Enc} + (ek_r + ek.c).H \stackrel{?}{=} G_{Enc}.r_{combined,r} + H.ek_r + Com_{ek}.c$$

8. Verifies the public key using $resp_{ek}$ from step 7:
   $$resp_{ek}.G_{Enc} \stackrel{?}{=} T_{pk} + pk.c$$
   $$\implies (ek_r + ek.c).G_{Enc} \stackrel{?}{=} G_{Enc}.ek_r + pk.c$$

9. Verifier finally checks the Bulletproofs proof that verifies all range constraints.
