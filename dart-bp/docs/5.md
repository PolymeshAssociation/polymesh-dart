# Sender/Receiver Affirmations

## Affirmations
Once has settlement has been posted on chain, sender/receiver send their agreement along with a proof which proves that they are the party to that leg, and they have done a correct account state transition, invalidating previous state by submitting the nullifier and their revealing their new state. Note that they don't point to their old state in the accumulator.

For an account state to transition from $State_{old}$ to $State_{new}$,
- secret key $sk$, identity $id$ and asset-id $at$ should not change
- nullifier should be revealed as $\rho^i.G_5$
- nullifier secret key $\rho^i$ and randomness $s^i$ should change as: $\rho^{i+1} = \rho.\rho^i$, $s^{i+1} = s.s^i$

Old state
$$
State_{old} = sk.G_{Aff} + balance.G_1 + counter.G_2 + at.G_3 + \rho.G_4 + \rho^i.G_5 + s.G_6 + s^i.G_7 + id.G_8 + ek.G_{Enc} \quad State_{old} \in \mathbb{G}_p
$$

New state
$$
State_{new} = sk.G_{Aff} + balance'.G_1 + counter'.G_2 + at.G_3 + \rho.G_4 + \rho^{i+1}.G_5 + s.G_6 + s^{i+1}.G_7 + id.G_8 + ek.G_{Enc} \quad State_{new} \in \mathbb{G}_p
$$

$balance$ might or might not be same as $balance'$ depending on who is affirming or what kind of transaction it is. $counter'$ might be same or 1 more or 1 less than $counter$ depending on the kind of transaction.
Also, it must be proved that the $State_{old}$ has same asset-id $at$ that is in the leg and has the public key for the same secret key as in the account.


For a single leg settlement of amount $v$:

1. When a sender affirms a leg, its new account state's balance should be $v$ less than the old account's and its counter should increase by 1.
2. When a receiver affirms a leg, its new account state's balance should be same as the old account's and its counter should increase by 1.
3. Sender can revert its affirmation by creating a new account state which has the balance increased by $v$ and counter decreased by 1.
4. Receiver can revert its affirmation by creating a new account state with counter decreased by 1.
5. Once a settlement has been confirmed, i.e. it cant be reverted,
    - sender can send a counter update txn to decrease it's counter by 1
    - receiver can send a claim funds txn to increases it's balance by $v$ and decrease it's counter by 1.

However, sender and receiver do have an option to send an irreversible affirmation and in that case, they don't change their counter, unless the settlement expires in which case the party who had sent, reverses it. 

Table describing changes to balance and counter for various transactions. Subscripts $_s$ and $_r$ refer to txn sent by sender and receiver respectively.

| Transaction Type     | Description                                       | Balance Change | Counter Change |
| -------------------- | ------------------------------------------------- | -------------- | -------------- |
| **$Affirm_s$**       | Sender affirms a leg of amount $v$                | **-v**         | **+1**         |
| **$Affirm_r$**       | Receiver affirms a leg                            | **0**          | **+1**         |
| **$Claim_r$**        | Receiver claims funds after settlement            | **+v**         | **-1**         |
| **$CntUpd_s$**       | Sender updates counter after settlement           | **0**          | **-1**         |
| **$Reverse_s$**      | Sender reverses their affirmation                 | **+v**         | **-1**         |
| **$Reverse_r$**      | Receiver reverses their affirmation               | **0**          | **-1**         |
| **$Irr$-$Affirm_s$** | Sender irreversibly affirms a leg of amount $v$   | **-v**         | **0**          |
| **$Irr$-$Affirm_r$** | Receiver irreversibly affirms a leg of amount $v$ | **+v**         | **0**          |

### Protocol
This section describes a unified account state transition proof which can handle all the above scenarios and parts of this proof can be omitted when balance doesn't change
Here the prover (sender/receiver) wants to prove following relations when sending affirmation for leg with amount $v$:

1. $State_{old} = sk.G_{Aff} + bal_0.G_1 + cnt_0.G_2 + at.G_3 + \rho.G_4 + \rho^i.G_5 + s.G_6 + s^i.G_7 + id.G_8 + ek.G_{Enc}$
2. Similarly, $State_{new} = sk.G_{Aff} + bal_1.G_1 + cnt_1.G_2 + at.G_3 + \rho.G_4 + \rho^{i+1}.G_5 + s.G_6 + s^{i+1}.G_7 + id.G_8 + ek.G_{Enc}$
3. $State_{old}$ exists in the accumulator (account curve tree)
4. $N = \rho^i.G_5$
5. $\rho^{i+1} = \rho.\rho^i$
6. $s^{i+1} = s.s^i$
7. If balance change is expected,
    - $bal_1 = bal_0 - v$ (for **Affirm_s**, **$Irr$-$Affirm_s$**)
    - $bal_1 = bal_0 + v$ (for **Claim_r**, **Reverse_s**, **$Irr$-$Affirm_r$**)
8. Counter change:
    - $cnt_1 = cnt_0 + 1$ (**Affirm_s**, **Affirm_r**)
    - $cnt_1 = cnt_0 - 1$ (**Claim_r**, **CntUpd_s**, **Reverse_s**, **Reverse_r**)
    - $cnt_1 = cnt_0$ (**$Irr$-$Affirm_s$**, **$Irr$-$Affirm_r$**)
9. Since the change to counter is public, relation for $State_{new}$ can be expressed as:
    - $State_{new} - G_2 = sk.G_{Aff} + bal_1.G_1 + cnt.G_2 + at.G_3 + \rho.G_4 + \rho^{i+1}.G_5 + s.G_6 + s^{i+1}.G_7 + id.G_8 + ek.G_{Enc}$ if counter increased by 1.
    - $State_{new} + G_2 = sk.G_{Aff} + bal_1.G_1 + cnt.G_2 + at.G_3 + \rho.G_4 + \rho^{i+1}.G_5 + s.G_6 + s^{i+1}.G_7 + id.G_8 + ek.G_{Enc}$ if counter decreased by 1.
    - $State_{new} = sk.G_{Aff} + bal_1.G_1 + cnt.G_2 + at.G_3 + \rho.G_4 + \rho^{i+1}.G_5 + s.G_6 + s^{i+1}.G_7 + id.G_8 + ek.G_{Enc}$ if counter remains same.
10. $bal_1 <= MAX\_BALANCE$ to avoid overflows
11. Prove that the party's account corresponds to the party in the leg. The account commitment contains $ek$ on $G_{Enc}$. The Bulletproof commitment includes both $ek$ and $ek^{-1}$ with the constraint $ek \cdot ek^{-1} = 1$, which allows $ek^{-1}$ to be used in ciphertext proofs. The structure of this proof depends on whether the asset-id is revealed to the verifier:
    - **Asset-id hidden**: Prove knowledge of $ek^{-1}$ and $at$ in $CT_{at} = Eph_{at} \cdot ek^{-1} + H \cdot at$, where $Eph_{at}$ is the party's ephemeral key for the asset-id ciphertext. $ek^{-1}$ comes from the Bulletproof commitment, $at$ is shared from the account state commitment proof.
    - **Asset-id revealed in this leg**: Prove knowledge of $ek^{-1}$ and $ek$ in $CT_s = Eph_s \cdot ek^{-1} + G_{Enc} \cdot ek$ (sender) or $CT_r = Eph_r \cdot ek^{-1} + G_{Enc} \cdot ek$ (receiver). $ek^{-1}$ comes from the Bulletproof commitment; $ek$ is shared from the account state commitment.
    - **Asset-id revealed but not in this leg**: Since $at$ is now public, prove knowledge of $ek^{-1}$ in $CT_{at} - H \cdot at = Eph_{at} \cdot ek^{-1}$. $ek^{-1}$ comes from the Bulletproof commitment.
12. If transition involves a balance change, prove that amount in $CT_v$ is $v$: prove knowledge of $ek^{-1}$ and $v$ in $CT_v = Eph_v \cdot ek^{-1} + H \cdot v$, where $Eph_v$ is the party's ephemeral key for the amount ciphertext. $ek^{-1}$ comes from the Bulletproof commitment; $v$ is shared from the Bulletproof balance commitment.

**Instance**: $State_{old_r}$, $State_{new}$, $N$, $Path_r$, $Root$, $CT_s$, $CT_r$, $CT_v$, $CT_{at}$ (a ciphertext when asset-id is hidden, or the revealed value $at$ otherwise), $Eph_s$, $Eph_r$, $Eph_v$, $Eph_{at}$ (only when asset-id is hidden), $EK_s$, $EK_r$, $G_{Aff}$, $G_{Enc}$, $G_i$, $H$, $H_i$, $\widetilde{G}$, $\widetilde{G}_i$, $\widetilde{H}$, $\widetilde{H}_i$

Here $G_{Enc}$ is the generator for the $ek$ component of the account commitment, and also serves as the key generator in ciphertexts. $H$ is the generator for plaintext values in ciphertexts.

**Witness**: $sk, ek, id, v, bal_0, bal_1, cnt, \rho, \rho^i, \rho^{i+1}, s, s^i, s^{i+1}, Path$ and additionally $at$ when asset-id is hidden. $ek^{-1}$ is derived from $ek$ and committed in the Bulletproof with constraint $ek \cdot ek^{-1} = 1$.

Above are the high level instance and witness values known before the protocol starts and don't include instance and witness created during protocol execution like the various blindings and $T$ values.

#### Prover
1. Prover first wants to prove knowledge of $sk, ek, at, bal_0, cnt, \rho, \rho^i, s, s^i, id$ in the relation $$State_{old} = sk.G_{Aff} + bal_0.G_1 + cnt_0.G_2 + at.G_3 + \rho.G_4 + \rho^i.G_5 + s.G_6 + s^i.G_7 + id.G_8 + ek.G_{Enc}$$
2. Similarly, prover wants to prove knowledge of $sk, ek, at, bal_1, cnt, \rho, \rho^{i+1}, s, s^{i+1}, id$ in $State_{new}$ from relation 9.
    - If counter increased by 1: $$State_{new} - G_2 = sk.G_{Aff} + bal_1.G_1 + cnt_0.G_2 + at.G_3 + \rho.G_4 + \rho^{i+1}.G_5 + s.G_6 + s^{i+1}.G_7 + id.G_8 + ek.G_{Enc}$$
    - If counter decreased by 1: $$State_{new} + G_2 = sk.G_{Aff} + bal_1.G_1 + cnt_0.G_2 + at.G_3 + \rho.G_4 + \rho^{i+1}.G_5 + s.G_6 + s^{i+1}.G_7 + id.G_8 + ek.G_{Enc}$$
    - If counter remains same: $$State_{new} = sk.G_{Aff} + bal_1.G_1 + cnt_0.G_2 + at.G_3 + \rho.G_4 + \rho^{i+1}.G_5 + s.G_6 + s^{i+1}.G_7 + id.G_8 + ek.G_{Enc}$$
3. Prover gets the path of the leaf $State_{old}$ as $Path$, an array of nodes (curve points), and randomizes it by adding blindings to all nodes. This will result in the leaf $State_{old}$ being transformed into $State_{old_r} = State_{old} + b_0.H_0$ and $Path$ transforms into $Path_r$. Prover enforces the constraints for curve tree membership using $Path, Path_r, b_i$.
4. Since prover knows the opening of $State_{old}$ and $b_0$, it can prove the knowledge of opening of $State_{old_r}$ as well. So relation 1 can be transformed as:
   $$
   sk.G_{Aff} + bal_0.G_1 + cnt_0.G_2 + at.G_3 + \rho.G_4 + \rho^i.G_5 + s.G_6 + s^i.G_7 + id.G_8 + ek.G_{Enc} + b_0.H_0 = State_{old_r}
   $$
5. Prover picks random $$sk_r, ek_r, at_r,  bal_{0_r},  cnt_r, \rho_r, \rho^i_r, \rho^{i+1}_r, s_r, s^i_r, s^{i+1}_r, id_r, b_{0_r} \in \mathbb{Z}_p$$ and creates:
   $$T_{State_{old_r}} = sk_r.G_{Aff} + ek_r.G_{Enc} +  bal_{0_r}.G_1 +  cnt_r.G_2 + at_r.G_3 + \rho_r.G_4 + $$
   $$\rho^i_r.G_5 + s_r.G_6 + s^i_r.G_7 + id_r.G_8 + b_{0_r}.H_0, \in \mathbb{G}_p$$
6. For the new state, create:
   $$T_{State_{new}} = sk_r.G_{Aff} + ek_r.G_{Enc} +  bal_{1_r}.G_1 +  cnt_r.G_2 + at_r.G_3 + \rho_r.G_4 + $$
   $$\rho^{i+1}_r.G_5 + s_r.G_6 + s^{i+1}_r.G_7 + id_r.G_8, \in \mathbb{G}_p$$

   Same blindings for $sk, ek, at, cnt, \rho, s, id$ are used in both old and new state since these values don't change. If balance doesn't change, also use same blinding so $bal_{0_r} = bal_{1_r}$.
7. For the correctness of nullifier, prover creates:
   $$
   T_{null} = \rho^i_r.G_5, \in \mathbb{G}_p
   $$
8. For enforcing $\rho^{i+1} = \rho.\rho^i$, $s^{i+1} = s.s^i$ and $ek \cdot ek^{-1} = 1$, prover sets up the constraints in Bulletproof. The first two are multiplications for randomness evolution, the third links $ek$ (committed in the account state) with $ek^{-1}$ (needed for ciphertext proofs). Prover uses Sigma protocol (Chaum Pedersen) to prove equality of committed values in Bulletproof's commitment and in the state commitments. Let $C_{BP_{\rho,s}}$ be the commitment:
     $$C_{BP_{\rho,s}} = b'.H_0 + \rho.H_1 + \rho^i.H_2 + \rho^{i+1}.H_3 + s.H_4 + s^i.H_5 + s^{i+1}.H_6 + ek.H_7 + ek^{-1}.H_8, \in \mathbb{G}_p$$
9. If the balance changes, prover sets up constraint in Bulletproof based on transaction type:
    - If balance change is $-v$: $bal_1 = bal_0 - v$
    - If balance change is $+v$: $bal_1 = bal_0 + v$

   Let $C_{BP_{bal}}$ be the commitment:
   $$
   C_{BP_{bal}} = b''.H_0 + v.H_1 + bal_0.H_2 + bal_1.H_3, \in \mathbb{G}_p
   $$
10. Prover creates commitment for $C_{BP_{\rho,s}}$ by picking random $r_{BP_{\rho,s}}, ek^{-1}_r \in \mathbb{Z}_p$:
    $$
    T_{BP_{\rho,s}} = r_{BP_{\rho,s}}.H_0 + \rho_r.H_1 + \rho^i_r.H_2 + \rho^{i+1}_r.H_3 + s_r.H_4 + s^i_r.H_5 + s^{i+1}_r.H_6 + ek_r.H_7 + ek^{-1}_r.H_8, \in \mathbb{G}_p
    $$
11. Prover creates commitment for $C_{BP_{bal}}$ by picking random $v_r, r_{BP_{bal}} \in \mathbb{Z}_p$:
    $$
    T_{BP_{bal}} = r_{BP_{bal}}.H_0 + v_r.H_1 +  bal_{0_r}.H_2 +  bal_{1_r}.H_3, \in \mathbb{G}_p
    $$
12. For the leg-account link, prover computes $T_{link}$ using only shared blindings — no new random value is picked. The structure depends on whether asset-id is revealed:
    - **Asset-id hidden**: $T_{link} = ek^{-1}_r \cdot Eph_{at} + at_r \cdot H$, using $ek^{-1}_r$ from step 10 (Bulletproof commitment) and $at_r$ from step 5.
    - **Asset-id revealed in this leg**: $T_{link} = ek^{-1}_r \cdot Eph_s + ek_r \cdot G_{Enc}$ (sender) or $T_{link} = ek^{-1}_r \cdot Eph_r + ek_r \cdot G_{Enc}$ (receiver), using $ek^{-1}_r$ from step 10 and $ek_r$ shared with the account state commitment (step 5).
    - **Asset-id revealed but not in this leg**: $T_{link} = ek^{-1}_r \cdot Eph_{at}$, using $ek^{-1}_r$ from step 10 ($at$ is public, no blinding needed).
13. If balance changes, prover computes $T_v = ek^{-1}_r \cdot Eph_v + v_r \cdot H$, using $ek^{-1}_r$ from step 10 and $v_r$ from the Bulletproof balance commitment.
15. Prover hashes the following to create challenge $c$ as:
    $$c = Hash(State_{old_r}, State_{new}, N, Path_r, Root, txn\_type, CT_s, CT_r, CT_v, CT_{at}, Eph_s (\text{or } Eph_r), Eph_{at} (\text{or } at \text{ if revealed}), Eph_v, EK_s (\text{or } EK_r), C_{BP_{\rho,s}}, C_{BP_{bal}}, $$
    $$T_{State_{old_r}}, T_{State_{new}}, T_{null}, T_{BP_{\rho,s}}, T_{BP_{bal}}, T_{link}, T_v)$$
16. Prover creates responses for $resp_{state_{old_r}}$. When asset-id is hidden, $at$ is also a witness (11 elements). When revealed, $at$ is public and its generator $G_3$ is dropped (10 elements):
    - **Asset-id hidden**:
    $$[\begin{matrix}
    sk_r + sk.c, & ek_r + ek.c, & bal_{0_r} + bal_0.c, & cnt_r + cnt_0.c, & at_r + at.c, & \rho_r + \rho.c, & \rho^i_r + \rho^i.c, & s_r + s.c, & s^i_r + s^i.c, & id_r + id.c, & b_{0_r} + b_0.c\\
    \end{matrix}]$$
    - **Asset-id revealed** (no $at$ response; indices shift accordingly):
    $$[\begin{matrix}
    sk_r + sk.c, & ek_r + ek.c, & bal_{0_r} + bal_0.c, & cnt_r + cnt_0.c, & \rho_r + \rho.c, & \rho^i_r + \rho^i.c, & s_r + s.c, & s^i_r + s^i.c, & id_r + id.c, & b_{0_r} + b_0.c\\
    \end{matrix}]$$
17. Prover creates responses for $resp_{state_{new}}$. Most witnesses are shared ($\bot$). If balance changes, create response for $bal_1$. Using the same indexing as step 16:
    $$[\begin{matrix}
    \bot, & \bot, & bal_{1_r} + bal_1.c \text{ or } \bot, & \bot, & \bot, & \bot, & \rho^{i+1}_r + \rho^{i+1}.c, & \bot, & s^{i+1}_r + s^{i+1}.c, & \bot, & \bot\\
    \end{matrix}]$$
    (Indices match step 16: 0=$sk$, 1=$ek$, 2=$bal$, 3=$cnt$, 4=$at$ (hidden only), 5=$\rho$, 6=$\rho^i$, 7=$s$, 8=$s^i$, 9=$id$, 10=$b_0$)
18. Response for nullifier:
    $$resp_{null} = \rho^i_r + \rho^i.c$$
19. For relation $C_{BP_{\rho,s}}$, only the response for blinding $b'$ needs to be created:
    $resp_{b'} = r_{BP_{\rho,s}} + b'.c$
20. For relation $C_{BP_{bal}}$, only the response for blinding $b''$ and amount $v$ needs to be created:
    $resp_{b''} = r_{BP_{bal}} + b''.c$
    $resp_{v} = v_r + v.c$
21. No separate responses are created for $T_{link}$ or $T_v$ — all witnesses are shared:
    - $ek^{-1}$ response: reused from $resp_{BP_{\rho,s}}$ (the Bulletproof response at witness index 8).
    - $at$ response (Case A, hidden): reused from $resp_{state_{old_r}}[4]$.
    - $ek$ response (Case B, revealed in this leg): reused from $resp_{state_{old_r}}[1]$ (shared with account commitment).
    - $v$ response: reused from $resp_v$ (balance Bulletproof response).
22. The proof is
    $$
    (C_{BP_{\rho,s}}, C_{BP_{bal}}, State_{old_r}, State_{new}, Path_r, N, T_{State_{old_r}}, T_{State_{new}}, $$
    $$T_{null}, T_{BP_{\rho,s}}, T_{BP_{bal}}, T_{link}, T_v, resp_{state_{old_r}}, resp_{state_{new}}, resp_{null}, $$
    $$resp_{b'}, resp_{b''}, resp_{v})
    $$


#### Verifier

1. Verifier knows the transaction type $txn\_type$ which determines counter direction and balance change expectations.
2. Verifier hashes the following to create challenge $c$ as:
   $$
   c = Hash(State_{old_r}, State_{new}, N, Path_r, Root, txn\_type, C_{BP_{\rho,s}}, C_{BP_{bal}}, $$
   $$T_{State_{old_r}}, T_{State_{new}}, T_{null}, T_{BP_{\rho,s}}, T_{BP_{bal}}, T_{link}, T_v)
   $$
3. Verifier enforces the Bulletproof constraints for the 2 multiplications: $\rho^{i+1} = \rho.\rho^i$ and $s^{i+1} = s.s^i$, and the inverse constraint $ek \cdot ek^{-1} = 1$.
4. Verifier enforces the Bulletproof constraint for the balance change based on $txn\_type$:
    - If $txn\_type \in \{Affirm_s, Irr-Affirm_s\}$: $bal_1 = bal_0 - v$
    - If $txn\_type \in \{Claim_r, Reverse_s, Irr-Affirm_r\}$: $bal_1 = bal_0 + v$
5. Verifier enforces the Bulletproof constraints for curve tree membership to verify that $Path_r$ leads to $Root$.
6. Verifies correctness of $State_{old_r}$. When asset-id is hidden, $G_3$ is part of the commitment and all 11 responses are used. When asset-id is revealed, $at \cdot G_3$ is subtracted from $State_{old_r}$ (since $at$ is public) and $G_3$ is dropped from the generator list, shifting indices 4+ down by one:
   - **Asset-id hidden** (indices 0–10 as in step 16 hidden case):
     $$resp_{state_{old_r}}[0].G_{Aff} + resp_{state_{old_r}}[1].G_{Enc} + resp_{state_{old_r}}[2].G_1 + resp_{state_{old_r}}[3].G_2 + resp_{state_{old_r}}[4].G_3 + $$
     $$resp_{state_{old_r}}[5].G_4 + resp_{state_{old_r}}[6].G_5 + resp_{state_{old_r}}[7].G_6 + resp_{state_{old_r}}[8].G_7 + $$
     $$resp_{state_{old_r}}[9].G_8 + resp_{state_{old_r}}[10].H_0 \stackrel{?}{=} T_{State_{old_r}} + State_{old_r}.c$$
   - **Asset-id revealed** (indices 0–9 as in step 16 revealed case; target adjusted by subtracting $at.G_3$):
     $$resp_{state_{old_r}}[0].G_{Aff} + resp_{state_{old_r}}[1].G_{Enc} + resp_{state_{old_r}}[2].G_1 + resp_{state_{old_r}}[3].G_2 + $$
     $$resp_{state_{old_r}}[4].G_4 + resp_{state_{old_r}}[5].G_5 + resp_{state_{old_r}}[6].G_6 + resp_{state_{old_r}}[7].G_7 + $$
     $$resp_{state_{old_r}}[8].G_8 + resp_{state_{old_r}}[9].H_0 \stackrel{?}{=} T_{State_{old_r}} + (State_{old_r} - at.G_3).c$$

7. Verifies correctness of $State_{new}$ by checking (reusing responses for shared witnesses from $resp_{state_{old_r}}$). The same hidden/revealed distinction applies as in step 6. Letting $D_{new}$ denote the adjusted $State_{new}$ (i.e., $State_{new}$ when hidden, or $State_{new} - at.G_3$ when revealed):
    - If counter increased by 1: target is $D_{new} - G_2$
    - If counter decreased by 1: target is $D_{new} + G_2$
    - If counter remains same: target is $D_{new}$

    The generator-to-response mapping follows step 16 (with $G_3$ and its response present or absent depending on whether asset-id is hidden or revealed). If balance doesn't change, $resp_{state_{new}}[2]$ is $\bot$ and verifier uses $resp_{state_{old_r}}[2]$ for both old and new balance responses. Similarly, $s$ (index 7) is shared between old and new states.

8. Verifies correctness of nullifier by checking:
   $$resp_{null}.G_5 \stackrel{?}{=} T_{null} + N.c$$
   $$\implies (\rho^i_r + \rho^i.c).G_5 \stackrel{?}{=} \rho^i_r.G_5 + N.c$$

9. Verifier checks response for $C_{BP_{\rho,s}}$ as:
   $$resp_{b'}.H_0 + resp_{state_{old_r}}[5].H_1 + resp_{state_{old_r}}[6].H_2 + resp_{state_{new}}[6].H_3 + resp_{state_{old_r}}[7].H_4 + $$
   $$resp_{state_{old_r}}[8].H_5 + resp_{state_{new}}[8].H_6 + resp_{state_{old_r}}[1].H_7 + resp_{BP_{\rho,s}}[8].H_8 \stackrel{?}{=} T_{BP_{\rho,s}} + C_{BP_{\rho,s}}.c$$
   $$\implies (r_{BP_{\rho,s}} + b'.c).H_0 + (\rho_r + \rho.c).H_1 + (\rho^i_r + \rho^i.c).H_2 + (\rho^{i+1}_r + \rho^{i+1}.c).H_3 + (s_r + s.c).H_4 + $$
   $$(s^i_r + s^i.c).H_5 + (s^{i+1}_r + s^{i+1}.c).H_6 + (ek_r + ek.c).H_7 + (ek^{-1}_r + ek^{-1}.c).H_8 \stackrel{?}{=} $$
   $$r_{BP_{\rho,s}}.H_0 + \rho_r.H_1 + \rho^i_r.H_2 + \rho^{i+1}_r.H_3 + s_r.H_4 + s^i_r.H_5 + s^{i+1}_r.H_6 + ek_r.H_7 + ek^{-1}_r.H_8 + C_{BP_{\rho,s}}.c$$
   Note: indices 5, 6 refer to $\rho, \rho^i$ responses from $resp_{state_{old_r}}$ (hidden asset-id case). When asset-id is revealed, these shift down by one (indices 4, 5).

10. Verifier checks response for $C_{BP_{bal}}$ as:
    $$resp_{b''}.H_0 + resp_v.H_1 + resp_{state_{old_r}}[1].H_2 + resp_{state_{new}}[1].H_3 \stackrel{?}{=} T_{BP_{bal}} + C_{BP_{bal}}.c$$
    $$\implies (r_{BP_{bal}} + b''.c).H_0 + (v_r + v.c).H_1 + ( bal_{0_r} + bal_0.c).H_2 + ( bal_{1_r} + bal_1.c).H_3 \stackrel{?}{=} r_{BP_{bal}}.H_0 + $$
    $$v_r.H_1 +  bal_{0_r}.H_2 +  bal_{1_r}.H_3 + C_{BP_{bal}}.c$$

11. Verifier checks the leg-account link using shared responses — no separate response was created for $T_{link}$:
    - **Asset-id hidden**: using $resp_{BP_{\rho,s}}[8]$ for $ek^{-1}$ and $resp_{state_{old_r}}[4]$ for $at$:
      $$resp_{BP_{\rho,s}}[8] \cdot Eph_{at} + resp_{state_{old_r}}[4] \cdot H \stackrel{?}{=} T_{link} + CT_{at} \cdot c$$
    - **Asset-id revealed in this leg** (sender): using $resp_{BP_{\rho,s}}[8]$ for $ek^{-1}$ and $resp_{state_{old_r}}[1]$ for $ek$:
      $$resp_{BP_{\rho,s}}[8] \cdot Eph_s + resp_{state_{old_r}}[1] \cdot G_{Enc} \stackrel{?}{=} T_{link} + CT_s \cdot c$$
      (receiver): $$resp_{BP_{\rho,s}}[8] \cdot Eph_r + resp_{state_{old_r}}[1] \cdot G_{Enc} \stackrel{?}{=} T_{link} + CT_r \cdot c$$
    - **Asset-id revealed but not in this leg**: using $resp_{BP_{\rho,s}}[8]$ for $ek^{-1}$; $at$ is public so the target is adjusted:
      $$resp_{BP_{\rho,s}}[8] \cdot Eph_{at} \stackrel{?}{=} T_{link} + (CT_{at} - H \cdot at) \cdot c$$

12. If balance changes, verifier checks the amount ciphertext using shared responses:
    $$resp_{BP_{\rho,s}}[8] \cdot Eph_v + resp_v \cdot H \stackrel{?}{=} T_v + CT_v \cdot c$$
    ($resp_{BP_{\rho,s}}[8]$ is the $ek^{-1}$ response from the Bulletproof commitment; $resp_v$ is the $v$ response from the Bulletproof balance commitment)

14. Verifier finally checks the Bulletproofs proof for all constraints i.e. curve tree membership, multiplications, inverse relation, and balance change

## Proof of balance

An investor can be asked for proof of balance of certain account (of certain asset-id). 
The requester of this proof might be the auditor who has visibility of the pending settlements, or it 
might be someone else who the investor should prove that the sum of its finalized balance and balance 
in pending settlements is certain value. The prover's public key, asset-id and balance are revealed to the verifier. 
The investor also reveals the nullifier of the account to prove that it is using the latest account state as the verifier 
can check that nullifier has not been revealed. An improvement could be to prove knowledge of an unrevealed nullifier using a 
proof of non-membership in the nullifier set (nullifier set has to be an accumulator). 
The prover should share the index of account commitment in tree so verifier can efficiently fetch the commitment and compare. 
If its not possible then do a membership proof. Prover could hide the commitment by randomizing it with a new blinding ($State' = State + B.r'$)

There are 2 variants of this proof:

1. **Proof of balance with auditor**: The verifier is an auditor who already has visibility of the account state including the counter. The prover only needs to prove correctness of the public key and nullifier against the account commitment.
2. **Proof of balance with non-auditor**: The verifier is an arbitrary party who may not have visibility of pending transactions. The prover must prove the counter equals the number of legs provided, prove their role in each leg, and prove the sum of pending amounts.

### Proof of balance with auditor

In this protocol, the verifier is an auditor who already has access to the account commitment and knows the counter value. The prover must prove:

1. The secret key $sk$ in the account commitment corresponds to the revealed public key $PK_{Aff}$
2. The nullifier $N$ is correctly derived from $\rho^i$ in the account commitment

**Instance**: $State, PK_{Aff}, at, balance, counter, id, N, G_{Aff}$

**Witness**: $sk, \rho, \rho^i, s, s^i, ek$

#### Prover

1. Prover wants to prove knowledge of $\rho, \rho^i, s, s^i, ek$ in the relation:
   $$\rho.G_4 + \rho^i.G_5 + s.G_6 + s^i.G_7 + ek.G_{Enc} = State - PK_{Aff} - balance.G_1 - counter.G_2 - at.G_3 - id.G_8$$
   where the RHS is public.
2. Prover wants to prove knowledge of $\rho^i$ in $N = \rho^i.G_5$.
3. Prover wants to prove knowledge of $sk$ in $PK_{Aff} = sk.G_{Aff}$.
4. Prover picks random $\rho_r, \rho^i_r, s_r, s^i_r, ek_r, sk_r \in \mathbb{Z}_p$ and creates:
   $$T_{acc} = \rho_r.G_4 + \rho^i_r.G_5 + s_r.G_6 + s^i_r.G_7 + ek_r.G_{Enc}, \in \mathbb{G}_p$$
   $$T_{null} = \rho^i_r.G_5, \in \mathbb{G}_p$$
   $$T_{pk} = sk_r.G_{Aff}, \in \mathbb{G}_p$$
5. Prover hashes the following to create challenge $c$ as:
   $$c = Hash(State, PK_{Aff}, at, balance, counter, id, N, T_{acc}, T_{null}, T_{pk})$$
6. Prover creates responses:
   $$resp_{acc} = [\begin{matrix}
   \rho_r + \rho.c, & \rho^i_r + \rho^i.c, & s_r + s.c, & s^i_r + s^i.c, & ek_r + ek.c\\
   \end{matrix}]$$
   $$resp_{pk} = sk_r + sk.c$$
7. The proof is
   $$(N, T_{acc}, T_{null}, T_{pk}, resp_{acc}, resp_{pk})$$

#### Verifier

1. Verifier hashes the following to create challenge $c$ as:
   $$c = Hash(State, PK_{Aff}, at, balance, counter, id, N, T_{acc}, T_{null}, T_{pk})$$
2. Verifies correctness of account state by checking:
   $$resp_{acc}[0].G_4 + resp_{acc}[1].G_5 + resp_{acc}[2].G_6 + resp_{acc}[3].G_7 + resp_{acc}[4].G_{Enc} \stackrel{?}{=} T_{acc} + (State - PK_{Aff} - balance.G_1 - counter.G_2 - at.G_3 - id.G_8).c$$
   $$\implies (\rho_r + \rho.c).G_4 + (\rho^i_r + \rho^i.c).G_5 + (s_r + s.c).G_6 + (s^i_r + s^i.c).G_7 + (ek_r + ek.c).G_{Enc} \stackrel{?}{=} \rho_r.G_4 + \rho^i_r.G_5 + $$
   $$s_r.G_6 + s^i_r.G_7 + ek_r.G_{Enc} + (State - PK_{Aff} - balance.G_1 - counter.G_2 - at.G_3 - id.G_8).c$$
3. Verifies correctness of nullifier by checking:
   $$resp_{acc}[1].G_5 \stackrel{?}{=} T_{null} + N.c$$
   $$\implies (\rho^i_r + \rho^i.c).G_5 \stackrel{?}{=} \rho^i_r.G_5 + N.c$$
4. Verifies knowledge of secret key by checking:
   $$resp_{pk}.G_{Aff} \stackrel{?}{=} T_{pk} + PK_{Aff}.c$$
   $$\implies (sk_r + sk.c).G_{Aff} \stackrel{?}{=} sk_r.G_{Aff} + PK_{Aff}.c$$

### Proof of balance with non-auditor

In this protocol, the verifier is an arbitrary party who may not have visibility of the pending transactions. The prover provides all pending leg encryptions and must additionally prove:

1. The counter equals the number of legs provided
2. The prover has the correct role (sender or receiver) in each leg and the asset-id in each leg matches the account's asset-id. Since the asset-id is revealed, for each leg $k$ the twisted Elgamal ciphertext is $CT_{at_k} = Eph_k \cdot ek^{-1} + at \cdot H$. The prover can prove this by showing $CT_{at_k} - at.H = Eph_k \cdot ek^{-1}$ where $Eph_k$ is the ephemeral key for the prover's role in the leg.
3. The sum of amounts in legs where prover is sender equals the declared pending sent amount. Let $S$ be the leg indices where prover is sender. The prover proves $\sum_{k \in S} CT_{v_k} - pending\_sent.H = (\sum_{k \in S} Eph_{s_k}) \cdot ek^{-1}$
4. The sum of amounts in legs where prover is receiver equals the declared pending received amount. Let $R$ be the leg indices where prover is receiver. The prover proves $\sum_{k \in R} CT_{v_k} - pending\_recv.H = (\sum_{k \in R} Eph_{r_k}) \cdot ek^{-1}$

**Instance**: $State, PK_{Aff}, PK_{Enc}, at, balance, counter, id, N, Leg_k, pending\_sent, pending\_recv, G_{Aff}, G_{Enc}, H$

**Witness**: $sk, ek, ek^{-1}, \rho, \rho^i, s, s^i$

Here $Leg_k$ denotes the set of leg encryptions for each leg, including ciphertexts and ephemeral keys. $PK_{Enc} = ek.G_{Enc}$ is the encryption public key which is subtracted as public in this version.

#### Prover

1. Prover wants to prove knowledge of $\rho, \rho^i, s, s^i$ in the account commitment relation (with $PK_{Aff}$ and $PK_{Enc}$ subtracted as public):
   $$\rho.G_4 + \rho^i.G_5 + s.G_6 + s^i.G_7 = State - PK_{Aff} - PK_{Enc} - balance.G_1 - counter.G_2 - at.G_3 - id.G_8$$
2. Prover wants to prove knowledge of $\rho^i$ in $N = \rho^i.G_5$ and $sk$ in $PK_{Aff} = sk.G_{Aff}$.
3. Prover proves knowledge of $ek$ in $PK_{Enc} = ek.G_{Enc}$ and $ek^{-1}$ in $G_{Enc} = PK_{Enc}.ek^{-1}$. The second relation proves $ek^{-1}$ is the true inverse of $ek$ since $PK_{Enc}.ek^{-1} = G_{Enc}.ek.ek^{-1} = G_{Enc}$. Both share the same blinding for $ek^{-1}$ so the response can be reused across all ciphertext proofs.
4. For each leg $k$, prove knowledge of $ek^{-1}$ in asset ID ciphertext:
   $$CT_{at_k} - at.H = ek^{-1} \cdot Eph_k$$
5. Let $S$ be the set of leg indices where prover is sender. Prove knowledge of $ek^{-1}$ in the sum of sent amounts:
   $$\sum_{k \in S} CT_{v_k} - pending\_sent.H = ek^{-1} \cdot \sum_{k \in S} Eph_{s_k}$$
6. Let $R$ be the set of leg indices where prover is receiver. Prove knowledge of $ek^{-1}$ in the sum of received amounts:
   $$\sum_{k \in R} CT_{v_k} - pending\_recv.H = ek^{-1} \cdot \sum_{k \in R} Eph_{r_k}$$
7. Prover picks random $\rho_r, \rho^i_r, s_r, s^i_r, sk_r, ek_r, ek^{-1}_r \in \mathbb{Z}_p$ and creates:
   $$T_{acc} = \rho_r.G_4 + \rho^i_r.G_5 + s_r.G_6 + s^i_r.G_7$$
   $$T_{null} = \rho^i_r.G_5$$
   $$T_{pk} = sk_r.G_{Aff}$$
   $$T_{EK} = ek_r.G_{Enc}$$
   $$T_{G_{Enc}} = ek^{-1}_r.PK_{Enc}$$
   For each leg $k$, creates $T_{at_k} = ek^{-1}_r \cdot Eph_k$  (where $Eph_k$ is the ephemeral key for the prover's role)
   $$T_{sent} = ek^{-1}_r \cdot \sum_{k \in S} Eph_{s_k}$$ (if $S$ is not empty)
   $$T_{recv} = ek^{-1}_r \cdot \sum_{k \in R} Eph_{r_k}$$ (if $R$ is not empty)
8. Prover hashes the following to create challenge $c$ as:
   $$c = Hash(State, PK_{Aff}, PK_{Enc}, at, balance, counter, id, N, Leg_k, pending\_sent, pending\_recv, $$
   $$T_{acc}, T_{null}, T_{pk}, T_{EK}, T_{G_{Enc}}, T_{at_k}, T_{sent}, T_{recv})$$
9. Prover creates responses for $\rho, \rho^i, s, s^i$:
    $$resp_{acc} = [\begin{matrix}
    \rho_r + \rho.c, & \rho^i_r + \rho^i.c, & s_r + s.c, & s^i_r + s^i.c\\
    \end{matrix}]$$
    $$resp_{pk} = sk_r + sk.c$$
    $$resp_{EK} = ek_r + ek.c$$
    $$resp_{G_{Enc}} = ek^{-1}_r + ek^{-1}.c$$
    Note that $resp_{G_{Enc}}$ is the response for $ek^{-1}$, which is reused to verify all the ciphertext relations.
10. The proof is
    $$(N, T_{acc}, T_{null}, T_{pk}, T_{EK}, T_{G_{Enc}}, T_{at_k}, T_{sent}, T_{recv}, resp_{acc}, resp_{pk}, resp_{EK}, resp_{G_{Enc}})$$

#### Verifier

1. Verifier reconstructs the challenge $c$ as
   $$c = Hash(State, PK_{Aff}, PK_{Enc}, at, balance, counter, id, N, Leg_k, pending\_sent, pending\_recv, $$
   $$T_{acc}, T_{null}, T_{pk}, T_{EK}, T_{G_{Enc}}, T_{at_k}, T_{sent}, T_{recv})$$
2. Verifier checks that counter equals number of legs provided.
3. Verifies correctness of account state by checking:
   $$resp_{acc}[0].G_4 + resp_{acc}[1].G_5 + resp_{acc}[2].G_6 + resp_{acc}[3].G_7 \stackrel{?}{=} $$
   $$T_{acc} + (State - PK_{Aff} - PK_{Enc} - balance.G_1 - counter.G_2 - at.G_3 - id.G_8).c$$
   $$\implies (\rho_r + \rho.c).G_4 + (\rho^i_r + \rho^i.c).G_5 + (s_r + s.c).G_6 + (s^i_r + s^i.c).G_7 \stackrel{?}{=} $$
   $$\rho_r.G_4 + \rho^i_r.G_5 + s_r.G_6 + s^i_r.G_7 + (State - PK_{Aff} - PK_{Enc} - balance.G_1 - counter.G_2 - at.G_3 - id.G_8).c$$
4. Verifies correctness of nullifier by checking:
   $$resp_{acc}[1].G_5 \stackrel{?}{=} T_{null} + N.c$$
   $$\implies (\rho^i_r + \rho^i.c).G_5 \stackrel{?}{=} \rho^i_r.G_5 + N.c$$
   Note that verifier uses $resp_{acc}[1]$ for checking both the account commitment and nullifier, proving that the same $\rho^i$ is used in both relations.
5. Verifies knowledge of secret key by checking:
   $$resp_{pk}.G_{Aff} \stackrel{?}{=} T_{pk} + PK_{Aff}.c$$
6. Verifies knowledge of encryption key $ek$ by checking:
   $$resp_{EK}.G_{Enc} \stackrel{?}{=} T_{EK} + PK_{Enc}.c$$
7. Verifies that $ek^{-1}$ is the inverse of $ek$ by checking:
   $$resp_{G_{Enc}}.PK_{Enc} \stackrel{?}{=} T_{G_{Enc}} + G_{Enc}.c$$
   $$\implies (ek^{-1}_r + ek^{-1}.c).PK_{Enc} \stackrel{?}{=} ek^{-1}_r.PK_{Enc} + G_{Enc}.c$$
   Since $PK_{Enc} = ek.G_{Enc}$, this proves $ek.ek^{-1} = 1$.
8. For each leg $k$, verifier checks asset-id correctness using the $ek^{-1}$ response:
   $$resp_{G_{Enc}} \cdot Eph_k \stackrel{?}{=} T_{at_k} + (CT_{at_k} - at.H).c$$
   $$\implies (ek^{-1}_r + ek^{-1}.c) \cdot Eph_k \stackrel{?}{=} ek^{-1}_r \cdot Eph_k + (CT_{at_k} - at.H).c$$
9. If prover is sender in any leg, verifier checks the total sent amount:
   $$resp_{G_{Enc}} \cdot (\sum_{k \in S} Eph_{s_k}) \stackrel{?}{=} T_{sent} + (\sum_{k \in S} CT_{v_k} - pending\_sent.H).c$$
   $$\implies (ek^{-1}_r + ek^{-1}.c) \cdot (\sum_{k \in S} Eph_{s_k}) \stackrel{?}{=} ek^{-1}_r \cdot (\sum_{k \in S} Eph_{s_k}) + (\sum_{k \in S} CT_{v_k} - pending\_sent.H).c$$
10. If prover is receiver in any leg, verifier checks the total received amount:
   $$resp_{G_{Enc}} \cdot (\sum_{k \in R} Eph_{r_k}) \stackrel{?}{=} T_{recv} + (\sum_{k \in R} CT_{v_k} - pending\_recv.H).c$$
   $$\implies (ek^{-1}_r + ek^{-1}.c) \cdot (\sum_{k \in R} Eph_{r_k}) \stackrel{?}{=} ek^{-1}_r \cdot (\sum_{k \in R} Eph_{r_k}) + (\sum_{k \in R} CT_{v_k} - pending\_recv.H).c$$
